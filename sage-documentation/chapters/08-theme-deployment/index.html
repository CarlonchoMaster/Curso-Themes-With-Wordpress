Theme deployment

When you’re ready to move your theme assets to production, you’ll need to generate the production ready assets:

# @ themes/theme-name/
$ yarn build:production

When your theme CSS and JS changes and you re-build them with yarn build:production, the filenames change due to revisioning. Instead of main.js and main.css, your filenames look similar to main_310b716c3997763c3838.js and main_310b716c3997763c3838.css.

The theme already knows which filenames to use thanks to the asset_path PHP helper function and @asset directive in Blade.

When you deploy your site you'll also need to make sure two things are covered:

  1. Run composer install from the theme directory on the remote server
  2. Production theme assets are either built on the server, or copied over from your local machine

If Composer is unavailable on your server, removing vendor/ from the .gitignore and committing this directory is the easiest solution.

Sage excludes the dist/ folder via .gitignore with your built theme assets by default. Ideally, you don’t ever want to commit compiled files to your Git repository.

When compiled assets are committed to your repository you're risking running into merge conflicts if multiple people are actively working on the same project. It's also a widely accepted best practice to keep compiled assets outside of the repo.

Since your assets don’t exist in your Git repository, you’ll need to compile them somehow during your deployment process. One option is to compile the assets locally and then copy them over. Another option is to compile them on the server during your deploy.

If this is not possible with your workflow, removing dist/ from the .gitignore file and committing it to your repository is the easiest solution, although it is never recommended.

Deploying with FTP

Using FTP (or even SFTP) for transferring files onto your server are considered a bad practice. There are several reasons why this is dangerous and should be avoided at all costs, including:

  • You cannot easily tell which version of the project is on the webserver
  • You cannot revert changes safely
  • You cannot tell who last made changes
  • You cannot easily see where changes have been made
  • Uploading files to the live web root will cause errors at times for active visitors on the site

“Sometimes I wish I could go back to live editing PHP files over FTP” — via @iamdevloper

That being said, I'm aware that many developers are still having to use FTP to get files up onto the server.

With the exception of the dist and vendor folders, everything else in the theme .gitignore file can be excluded from your file transfer. This comes down to making sure that you don't upload your node_modules directory.

The dg/ftp-deployment package might be worth a look. Also look into using Capistrano, which has been known to work with shared hosts that allow SSH access:

  • Deploying WordPress with Capistrano on GoDaddy Shared Hosting post by Jacob Williams
  • Deploying WordPress with Capistrano screencast by Scott Walkinshaw

FTP deployment services include Beanstalk, FTPloy, and DeployHQ.

Compile assets locally

Trellis allows you to deploy assets after compiling them locally using Ansible, and our Roots Example Project shows exactly how this is done in the deploy-hooks/build-before.yml file:

- name: Run yarn
  command: yarn
  connection: local
  args:
    chdir: "{{ project.local_path }}/web/app/themes/sage"

- name: Run build
  command: yarn build:production
  connection: local
  args:
    chdir: "{{ project.local_path }}/web/app/themes/sage"

- name: Copy project local files
  synchronize:
    src: "{{ project.local_path }}/web/app/themes/sage/dist"
    dest: "{{ deploy_helper.new_release_path }}/web/app/themes/sage"
    group: no
    owner: no
    rsync_opts: --chmod=Du=rwx,--chmod=Dg=rx,--chmod=Do=rx,--chmod=Fu=rw,--chmod=Fg=r,--chmod=Fo=r

- name: Install Composer dependencies
  command: composer install --no-ansi --no-dev --no-interaction --no-progress --optimize-autoloader --no-scripts
  args:
    chdir: "{{ deploy_helper.new_release_path }}/web/app/themes/sage"

During deployment of your theme, Trellis will run yarn, yarn build:production, and composer install in your theme directory. It also copies up the contents of themes/sage/dist/ to your server with rsync.

Compiling on server

Your server will need the same requirements as your local development machine in order to build the theme assets. That means:

  • Git
  • PHP >= 7.1.3
  • Composer
  • Node.js >= 6.9.1
  • Yarn

Then as part of your deploy process:

  • Run yarn build:production
  • Run composer install

Compiling on the server isn't the best solution for a resource intensive build process. If your theme contains a couple dozen images that get optimized during the build process, resources are being taken away from your webserver while you deploy your theme.

Deploying with a CI server

A continuous integration server will watch your repository for changes and then run an automated build process. By setting up continuous integration, you avoid the need to manually run deploys. Once a commit is merged into master, it's sent off to be deployed automatically.

This book won't get into detail about CI deploys, but you can get the idea of what's going on behind the scenes.

Circle CI deploys with Trellis

Below is an example circle.yml file for deploying a Sage theme onto a Trellis provisioned VPS.

machine:
 python:
   version: 3.5.0
 node:
   version: 6.9.5

dependencies:
 pre:
   - pip install ansible
   - cd trellis && ansible-galaxy install -r requirements.yml
   - yarn global add gulp bower

test:
  override:
    - cd site/web/app/themes/sage && yarn
    - cd site/web/app/themes/sage && yarn build:production
    - cd site/web/app/themes/sage && composer install

deployment:
 production:
   branch: master
   commands:
     - cd trellis && ansible-playbook deploy.yml --vault-password-file="~/.vault_pass" -e "site=roots-example-project.com env=production"

For an alternative Circle CI deploy approach, see the thread by Craig Dennis on Roots Discourse.
