Planning out the theme

You should aim to get started working on your project within WordPress as soon as possible. Rather than spending time building static HTML, CSS, and JS prototypes, start developing directly in the theme from the start.

A lot of developers are used to building in HTML first, then moving to WordPress. The modern front-end workflow found in Sage will prove to be extremely beneficial as soon as you launch your first Browsersync session.

Some static front-end generators such as Hugo are very attractive, but if at the end of the day you're going to end up in WordPress, you probably want to just start there to avoid duplicating any efforts.

I recently worked on an project where I built out a Hugo site with the same markup used in Sage and migrated over the build process. We knew that it was going to end up in WordPress eventually, but thought that by doing the initial prototype in Hugo that we could avoid conflicts with the existing WordPress site.

Since WordPress was the end goal, time was lost at the end of the day. We could have spent more time initially trying to figure out a better solution for dealing with varying environments, but instead that time (plus more) was spent dealing with the content in Hugo versus WordPress.

Keep site functionality outside of the theme

Make sure that you’re not registering any custom post types or taxonomies in your theme. Restrict the code in your theme to presentation and create site functionality plugins (or mu-plugins) for things such as:

  • Custom post types
  • Taxonomies
  • Shortcodes
  • Widgets
  • Admin changes (CSS & JS)
  • Miscellaneous functions not related to your theme

If you're used to using functions.php or other theme files for your code, consider creating a functionality plugin instead. A functionality plugin is kept in your plugins/ or mu-plugins/ directory on your WordPress installation.

Even if you don't think it's beneficial right now, spend the little extra time it'll take to create a functionality plugin. You'll be glad you did the next time, maybe years later, when it comes to create a new theme.

This can be as simple as placing a single file in the mu-plugins directory, located directly inside of the WordPress content directory alongside themes, plugins, and uploads:

<?php
// @ mu-plugins/demo.php

/**
 * Place ACF JSON in content directory
 */
add_filter('acf/settings/save_json', function ($path) {
    return WP_CONTENT_DIR . '/acf-json';
});
add_filter('acf/settings/load_json', function ($paths) {
    unset($paths[0]);
    $paths[] = WP_CONTENT_DIR . '/acf-json';
    return $paths;
});

An example mu-plugin setup from the Roots website:

Take a look at johnbillion/extended-cpts for a library that makes creating custom post types and taxonomies simple and beautiful:

<?php
/**
 * Team custom post type
 */
add_action('init', function () {
    register_extended_post_type('team', [
        'menu_icon' => 'dashicons-groups'
    ], [
        'singular' => 'Team Member',
        'plural' => 'Team Members',
        'slug' => 'team'
    ]);
});

/**
 * Team taxonomies
 */
add_action('init', function () {
    register_extended_taxonomy('team_category', 'team', [], [
        'singular' => 'Team Category',
        'plural' => 'Team Categories'
    ]);
});

Content and meta data checklist

Content precedes design. Design in the absence of content is not design, it's decoration.

— Jeffrey Zeldman

Do you have what you need to get started on your theme? It’s not necessary to have all of your content ready to go, but it’s a good idea to make a list of the various pages, sections, and components that will make up your site.

For example, if you’re going to be selling products on your website, you’ll want to have the product data and pictures ready so that you’ll be prepared when it comes to the point where you need to make your product templates.

If you’re building a portfolio site, you’ll want to have items with details prepared so that you can properly write your templates to account for the content that will be displayed.

Dummy theme data

There's a few options for getting some test content into your WordPress installation if you're looking for a way to quickly get started.

Typically I've just used the WordPress.org theme unit test and imported it via WP-CLI.

WP Test

WP Test is similar to the theme unit test, but offers the benefits of being able to watch activity on GitHub as well as more 'real' data instead of lorem ipsum.

The quickest way to get the WP Test data into your WordPress install:

# @ WordPress project root directory
$ curl -OL https://raw.githubusercontent.com/poststatus/wptest/master/wptest.xml
$ wp import wptest.xml --authors=create

WP-CLI

WP-CLI offers a wp post generate command, and you can use existing lorem ipsum services for the post content.

FakerPress

FakerPress has some advantages over the other options, including: random featured images, random meta creation, random terms generation, and user generation.

Starter content

As of WordPress 4.7 (December 2016), starter content can now be applied to themes that will be shown to users setting up their site for the first time. The starter content shows in the WordPress customizer and is controlled via add_theme_support('starter-content', [ ... ]);.

Starter content for themes in 4.7 from the Make WordPress Core blog provides the most documentation I could find at the time of writing.

The new Twenty Seventeen theme takes advantage of core-provided starter content.

My blog post, Using and Customizing WordPress Starter Content, shows some examples on how to use starter content.

Site map

Your site map doesn’t need to be completely figured out in order to start working on your theme, but you’ll want to know what’s going to be in your navigation menus — especially the primary navigation. An example primary navigation would be:

  • Home
  • Product
  • Pricing
  • Resources
  • Blog
  • About
  • Contact Us

Once you’ve got your primary navigation roughly figured out, go ahead and create the appropriate pages for them. You'll also need to create a new menu and assign the location to Primary Navigation, which is used in resources/views/partials/header.blade.php:

<nav class="nav-primary">
  @if (has_nav_menu('primary_navigation'))
    {!! wp_nav_menu(['theme_location' => 'primary_navigation', 'menu_class' => 'nav']) !!}
  @endif
</nav>

You’ll also want to update your Reading Settings in the WordPress admin to reflect having a static front page that isn’t a blog.

Navigation possibilities

Sage previously included support for the Bootstrap navbar component by default on the primary navigation. As of April 2015, the header template (resources/views/partials/header.php) uses a basic navigation by only applying the nav class to the menu.

The Bootstrap navbar component can be quickly added by following the instructions by Michael W. Delaney:

# @ themes/your-theme-name/
$ composer require mwdelaney/sage-bootstrap4-navwalker

<nav class="nav-primary">
  @if (has_nav_menu('primary_navigation'))
    {!! wp_nav_menu(['theme_location' => 'primary_navigation', 'menu_class' => 'nav', 'walker' => new wp_bootstrap4_navwalker())]) !!}
  @endif
</nav>

There are countless responsive navigation patterns that are probably a better fit:

  • Top nav (or “do nothing” approach)
  • Priority+
  • Off-canvas flyout
  • Carousel+

See more at Responsive Patterns.
